A06과 A07은 구현 레이어보다 한 단계 위에 존재하는 문제다. A06은 아키텍처 설계 단계에서 위협을 고려하지 않은 결과이고, A07은 사용자 신원을 검증하는 메커니즘 자체의 결함이다. 코드를 아무리 잘 짜도 이 두 영역이 취약하면 시스템은 무너진다.

## A06 — 안전하지 않은 설계 (Insecure Design)

안전하지 않은 설계와 안전하지 않은 구현은 다르다. 구현 결함은 코드를 수정하면 해결된다. 설계 결함은 시스템의 로직과 흐름 자체를 다시 그려야 한다. 2021년 신규 진입한 이 항목이 말하는 것은 코드 리뷰와 보안 테스트 이전 단계, 즉 소프트웨어 개발 생명주기(SDLC)의 초기 설계 과정에서 위협 모델링(Threat Modeling)이 이루어지지 않을 때 생기는 구조적 취약점이다.

쇼핑몰 결제 플로우는 이를 설명하는 가장 직관적인 사례다. 상품 결제 프로세스를 설계할 때, 가격 정보에 대한 위변조 가능성을 위협 모델에 포함시키지 않은 경우다. 사용자가 장바구니에 담긴 물품의 결제를 진행하는 중간에 패킷을 가로채 결제 금액 파라미터를 0으로 조작하고 전송했을 때, 백엔드 서버가 최종 결제 단계에서 원래 상품 가격과 대조하는 로직 없이 받은 금액을 그대로 처리한다면 공격자는 비용 지불 없이 상품을 구매할 수 있게 된다. 이것은 코드 버그가 아니다. 설계 단계에서 "결제 금액은 서버 측에서 독립적으로 계산·검증되어야 한다"는 보안 요구사항이 정의되지 않은 것이다.

위협 모델링은 설계 단계에서 공격자의 관점을 의도적으로 도입하는 작업이다. "공격자가 이 흐름에서 어떤 가정을 깰 수 있는가"를 다이어그램과 데이터 흐름으로 검토한다. 구현을 시작하기 전에 결함을 찾는 비용이 배포 후 패치 비용보다 몇 배 낮다는 것이 이 항목이 Top 10에 존재하는 이유다.

## A07 — 인증 실패 (Authentication Failures)

기존 '식별 및 인증 실패'에서 명칭이 단순화된 이 항목은 사용자의 신원을 증명하고 세션을 유지하는 메커니즘 전반의 결함을 다룬다. 인증은 시스템 진입의 첫 관문이기 때문에, 이 관문이 무너지면 이후의 모든 접근 제어가 의미를 잃는다.

**크리덴셜 스터핑과 세션 하이재킹** 이 가장 현실적인 공격 경로다. 다른 서비스에서 대규모로 유출된 수백만 건의 이메일·비밀번호 조합을 확보한 공격자가 봇넷을 이용해 타겟 애플리케이션의 로그인 API에 초당 수천 번의 인증 요청을 시도한다. 애플리케이션이 MFA를 요구하지 않거나 로그인 실패 횟수에 따른 계정 잠금(Rate Limiting) 정책을 구현하지 않으면, 동일한 비밀번호를 여러 서비스에서 재사용하는 사용자들의 계정이 대규모로 탈취된다. 사용자가 아무리 주의해도 애플리케이션이 방어하지 않으면 피해는 불가피하다.

세션 관리 결함은 더 조용하지만 지속적이다. 로그아웃 후 서버 측에서 세션 토큰을 명시적으로 폐기하지 않으면, 탈취된 세션 ID는 사용자가 로그아웃한 이후에도 재사용 가능한 상태로 유지된다. 이것은 코드 몇 줄의 문제지만, 설계 단계에서 "세션 종료 시 서버 측 무효화"를 명시적으로 요구사항으로 정의하지 않으면 빠뜨리기 쉬운 부분이다.

취약한 비밀번호 복구 흐름도 자주 간과된다. 보안 질문("어머니의 성함은?") 기반 복구는 사회공학적 정보 수집에 취약하고, 예측 가능한 토큰을 사용하는 이메일 기반 복구는 토큰 추측 공격에 노출된다.

MFA는 크리덴셜 탈취 공격의 대부분을 무력화한다. 비밀번호가 유출되어도 두 번째 인증 요소가 없으면 로그인이 불가능하기 때문이다. 표준화된 인증 프레임워크(OAuth 2.0, OIDC)의 채택은 직접 구현의 실수를 줄이는 가장 현실적인 방어다.
