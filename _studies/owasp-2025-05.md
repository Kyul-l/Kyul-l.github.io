A08, A09, A10은 흔히 "덜 중요한 항목"처럼 취급되지만, 실제 침해 사고에서 이 세 영역의 결함이 공격자의 횡이동(Lateral Movement)을 가능하게 하거나 침해 사실 자체를 은폐하는 역할을 한다. 마지막으로, 2025년이라는 시점에서 이 위협들이 AI·LLM 환경과 어떻게 교차하는지를 살펴본다.

## A08 — 소프트웨어·데이터 무결성 오류 (Software or Data Integrity Failures)

무결성 오류는 데이터나 코드가 악의적으로 변조되지 않았음을 보장하는 검증 장치가 없을 때 발생한다. 출처가 불분명한 플러그인을 무비판적으로 수용하거나, 객체 직렬화 보안이 미비할 때 시스템이 붕괴될 수 있다.

**안전하지 않은 역직렬화(Insecure Deserialization)** 는 Java, Python 같은 언어로 작성된 애플리케이션이 사용자 세션 데이터를 직렬화(Serialization)하여 쿠키에 저장해두었다가, 이를 서버에서 역직렬화(Deserialization)하여 사용할 때 무결성 검증(디지털 서명 확인)을 거치지 않는 경우다. 공격자가 쿠키 내의 직렬화 객체를 변조하여 원격 코드 실행(RCE) 페이로드를 삽입한 뒤 전송하면, 서버는 객체를 복원하는 과정에서 즉각적으로 악성 코드를 실행하게 된다. 역직렬화 자체가 취약점이 아니라, 역직렬화 이전에 출처와 무결성을 검증하지 않는 것이 취약점이다.

CI/CD 파이프라인의 무결성도 이 항목에 포함된다. 빌드 아티팩트나 컨테이너 이미지에 디지털 서명을 적용하고, 배포 전에 서명을 검증하는 프로세스가 없으면 A03(공급망 장애)과 유사한 경로로 변조된 바이너리가 운영 환경에 배포될 수 있다.

## A09 — 보안 로깅·알림 실패 (Security Logging and Alerting Failures)

로그가 있어도 실시간 알림과 연동되지 않으면 방치된 것이나 다름없다는 판단이 2025년에 '모니터링'에서 '알림(Alerting)'으로 명칭이 강화된 배경이다. 가시성(Visibility)만으로는 부족하고, 탐지 즉시 대응으로 이어지는 체계가 필요하다.

**침해 사고 인지 지연과 Log Injection** 관리자 계정에 대한 수천 번의 인증 실패가 발생하고 있는데도 모니터링 경보 규칙이 설정되어 있지 않아 공격이 수일간 계속되도록 방치되는 상황이 전형적이다. 더 나아가 애플리케이션이 사용자의 입력값을 그대로 로그 파일에 기록할 경우, 공격자가 입력 필드에 개행 문자(`\r\n`)를 포함한 위조 로그 문구를 주입(Log Injection)하여 실제 침해 흔적을 가리거나 관리자를 속이는 것이 가능해진다. 로그 자체가 신뢰할 수 없게 되는 상황이다.

효과적인 보안 로깅은 단순히 '무엇이 일어났는가'를 기록하는 것이 아니라, '무엇이 이상한가'를 즉시 탐지하고 대응 담당자에게 알리는 체계까지 포함한다.

## A10 — 예외 처리 오류 (Mishandling of Exceptional Conditions)

2025년 신규 항목으로, 시스템이 예상치 못한 환경적 장애·논리적 오류·예외 상황에 직면했을 때 어떻게 반응하는지에 초점을 맞춘다. 정상 흐름에서는 완벽하게 동작하는 코드가 예외 경로에서 보안 보장을 깨뜨린다.

**보안 제어의 Fail-Open** 은 가장 위험한 패턴이다. 권한 검증 로직이 외부 서비스(LDAP 또는 IAM 서버)와 통신하여 접근 허용 여부를 결정하는데, 네트워크 장애로 타임아웃이 발생하면 검증 함수가 예외를 반환한다. 이때 try-catch 구문에서 예외를 삼켜버리고(Swallow) 방어적으로 시스템을 차단(Fail-Closed)하는 대신 처리를 속행하여 무조건 접근을 허용해버리는 논리 오류다. 실제로 2025년 Open VSX 레지스트리의 보안 스캐너 파이프라인에서 스캐너가 과부하로 실패했을 때 이를 '위협 없음'으로 간주하여 악성 확장이 통과되는 Fail-Open 버그가 발견된 바 있다.

**불완전한 상태 롤백(Race Condition)** 금융 시스템에서 다단계 트랜잭션(출금 → 증액 → 로깅)을 처리하던 중 예외가 발생했을 때, 이전 단계로 완전히 롤백하지 않으면 잔고가 감소했으나 대상 계좌에는 증액이 반영되지 않는 등 치명적인 데이터 오염이 발생한다.

## AI·LLM 환경으로의 확장

2025년이라는 시점에서 이 위협들을 논할 때, AI와 LLM을 결합한 애플리케이션의 확산을 배제할 수 없다.

프롬프트 인젝션(Prompt Injection)은 A05(인젝션)와 동일한 논리적 근원을 가진다. AI 모델이 외부 신뢰할 수 없는 데이터(사용자 프롬프트, 웹 콘텐츠)를 검증 없이 수용하여 백엔드 명령어를 실행하는 것은, SQL 인젝션이 신뢰할 수 없는 입력을 쿼리로 실행하는 것과 구조적으로 같다. 오염된 AI 모델 가중치나 악의적인 라벨링 데이터를 외부에서 다운로드하는 위험은 A03(소프트웨어 공급망 장애)의 가장 최신 형태다. AI 시스템이 예외 상황에서 보안 검증을 우회하거나, 로그 없이 판단을 내리거나, 역직렬화된 컨텍스트를 무결성 검증 없이 신뢰하는 구조는 A08·A09·A10이 새로운 레이어에서 반복되는 것이다.

2025년 OWASP Top 10은 클라우드와 AI 시대를 살아가는 소프트웨어 엔지니어와 보안 실무자가 갖추어야 할 가장 명확한 이정표를 제시하고 있다. 단순히 취약점을 패치하는 것이 아니라, 설계 단계부터 복원력을 가진 생태계를 구축하는 것이 목표다.
