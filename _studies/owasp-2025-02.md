2025년 OWASP 상위 세 항목의 공통점은 코드 버그가 아니라는 것이다. A01은 인가 로직의 부재, A02는 운영 환경 설정의 방임, A03은 공급망 신뢰 관계의 악용에서 비롯된다. 코드 품질만으로는 해결되지 않는 영역이다.

## A01 — 취약한 접근 제어 (Broken Access Control)

접근 제어 취약점은 "이 사용자가 이 리소스에 접근할 수 있는가"를 백엔드가 제대로 검증하지 않을 때 발생한다. 인증(Authentication, 누구인가)과 인가(Authorization, 무엇을 할 수 있는가)를 혼동하는 개발 관행이 가장 큰 원인이다. 사용자가 올바른 JWT 토큰을 가지고 있더라도, 서버가 그 토큰의 소유자가 요청 대상 리소스의 정당한 소유자인지 확인하지 않으면 취약하다.

**BOLA(Broken Object Level Authorization)** 는 API 환경에서 가장 흔한 패턴이다. `GET /api/users/12345/orders` 엔드포인트에 접근할 때, 서버가 JWT를 검증하는 것으로 인가를 완료했다고 가정한다. 공격자가 URL의 `12345`를 `12346`으로 바꿔 전송하면, 서버는 해당 토큰이 유효하다는 이유로 타인의 주문 내역을 반환한다. '인증된 사용자'와 '해당 리소스에 접근 권한이 있는 사용자'를 서버가 구분하지 않은 것이다. 전통적인 DAST 도구는 HTTP 200 응답을 성공으로 판정하기 때문에 자동화된 스캔만으로는 탐지하기 매우 어렵다.

**SSRF(Server-Side Request Forgery)** 는 2025년에 A01의 하위 패턴으로 공식 편입되었다. 외부 URL 입력을 받아 서버가 대신 요청을 실행하는 기능—웹훅 처리, 외부 이미지 다운로드—에서 입력값 검증이 없으면 공격자는 서버를 내부 네트워크 접근의 중개자로 활용한다. `http://169.254.169.254/latest/meta-data/iam/security-credentials/` 같은 AWS 메타데이터 엔드포인트는 방화벽 외부에서는 도달할 수 없지만, 서버 자체가 요청을 실행하면 IAM 임시 자격 증명이 그대로 반환된다. 이것이 SSRF를 접근 제어 실패로 분류하는 이유다. 서버에 부여된 내부 접근 권한이 외부 입력에 의해 무단으로 행사된 것이다.

## A02 — 보안 설정 오류 (Security Misconfiguration)

테스트된 애플리케이션의 100%에서 어떤 형태로든 설정 오류가 발견된다는 것이 2025년 OWASP의 진단이다. 클라우드 스토리지 권한 설정 실수 한 줄이 수백만 건의 정보 유출로 이어진 사례는 이제 일상적 뉴스다.

**클라우드 스토리지 권한 개방** S3 버킷이나 Azure Blob Storage의 ACL이 실수로 `public-read`로 설정되면, 공격자는 자동화된 스캐너로 인터넷에 노출된 버킷을 탐색하고 민감한 고객 데이터, 데이터베이스 백업 파일, 내부 문서를 내려받는다. 설정 한 줄의 방임이 코드 수천 줄의 보안 노력을 무효화한다.

**장황한 에러 메시지를 통한 정보 유출(CWE-489, CWE-526)** 은 더 조용하지만 오래 지속되는 위협이다. 운영 서버에서 디버그 모드를 끄지 않으면, 공격자가 의도적으로 비정상적인 파라미터를 전송해 오류를 유발할 때 스택 트레이스·DB 쿼리 구조·환경 변수·사용 중인 라이브러리 버전이 응답에 그대로 담겨 반환된다. 이 정보들은 공격자에게 정찰(Reconnaissance) 단계를 크게 단축시켜 준다.

보안 설정 오류의 근본적인 문제는 '기본값의 위험성'이다. 프레임워크와 클라우드 서비스는 개발 편의를 위해 설계된 기본 설정으로 제공된다. 운영 환경에 배포할 때 그 기본값을 능동적으로 검토하고 강화하지 않는 한, 개발 편의 설정이 그대로 운영에 노출된다.

## A03 — 소프트웨어 공급망 장애 (Software Supply Chain Failures)

현대의 공격자들은 견고하게 방어된 타겟 애플리케이션을 직접 공격하는 대신, 그 애플리케이션이 의존하는 개발 생태계를 노린다. 이 전략이 효과적인 이유는 단순하다. 개발자는 내부 코드보다 외부 라이브러리를 훨씬 더 쉽게 신뢰하고 실행하기 때문이다.

**Dependency Confusion(의존성 혼동)** 은 이 신뢰를 직접 악용한다. 기업 내부 비공개 라이브러리와 동일한 이름으로 악성 코드가 포함된 패키지를 공개 레지스트리(npm, PyPI)에 더 높은 버전 번호로 등록한다. 내부 레지스트리보다 공개 레지스트리의 최신 버전을 우선하도록 CI/CD 파이프라인이 구성되어 있다면, 빌드 시스템이 빌드 타임에 스스로 악성 코드를 주입하는 결과가 된다. 2025년 발생한 Shai-Hulud npm 원 사건에서 악성 패키지는 개발자 환경에서 npm 토큰을 탈취하여 수백 개의 연관 패키지로 스스로 전파하는 연쇄 피해를 낳았다.

**업스트림 벤더 인프라 침해(SolarWinds 모델)** 는 규모가 다르다. 소프트웨어 공급업체의 빌드 시스템이 해킹되어 정상적으로 서명된 업데이트 파일에 백도어가 삽입된 채 배포된다. 수천 개의 기업·정부 기관이 공식 업데이트라는 이유로 이를 무조건 신뢰하고 설치함으로써 내부망이 순식간에 장악되는 연쇄 피해가 발생한다.

공급망 보안은 코드 레이어 아래의 문제다. SBOM(소프트웨어 부품 목록) 관리, 빌드 파이프라인의 무결성 서명, 의존성 레지스트리 설정 검토가 방어의 출발점이 된다.
